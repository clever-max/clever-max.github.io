基于动作录制的网课自动助手

基于动作录制的网课自动助手

人的懒惰是技术进步的第一驱动力。当我第三次在同一门网课里手动点击同一个”下一节”按钮时，我觉得是时候让计算机替我受这个罪了。

起因：一个刷课人的自我修养

事情要从一门让我昏昏欲睡的网课说起。每个视频都要手动点开、手动播放、手动等它放完，整套流程下来，我觉得自己不是在学知识，是在给鼠标做有氧运动。

于是我开始写自动化脚本。最初版本很粗糙 —— Selenium 驱动 Edge 浏览器，硬编码课程 URL，一个一个视频地点过去。能用，但仅限于雨课堂，仅限于一门课，换个课程就得改源码。这就像一把只能开一扇门的钥匙，换了门就废了。

那个旧版长这样：

每次换课都得进代码里改这些数字，比刷课还累。更糟糕的是，它严重依赖雨课堂的 DOM 结构 —— CSS 类名 .chapter、.activity__wrap 这些一旦改版，整个脚本当场去世。而且只能处理视频，遇到测验和文档直接跳过，仿佛在说”这题太难了我不做了”。

经历了三个版本的小修小补（修 StaleElement、加倍速播放、加暂停检测），我意识到这条路走不通 —— 靠硬编码定位元素，本质上就是在给网站打补丁，网站一更新你就得跟着改。

我需要一种更通用的方案。

思路转变：从”替你操作”到”学你操作”

旧方案的核心问题是：我替程序决定点哪里。每个按钮的 XPath 都是我写死的，网站一改就失效。

那如果反过来呢？让程序自己学会该点哪里？

于是有了新方案的核心思路 —— 录制-回放：

  1. 录制阶段：我像正常用户一样操作网课，浏览器扩展在后台记录每一步操作（点了哪里、输入了什么）
  2. 回放阶段：Python 程序读取录制结果，自动重放这些操作

这就像”宏录制”—— 你做一遍，程序照着做无数遍。换课？重新录制一遍就行。换平台？也能用，因为录制的是你的操作，不是我对网站的假设。

架构：一录一放，两段式设计

整个项目分成两个独立的部分：

录制端：Chrome 扩展

扩展基于 Manifest V3，核心就一个 content.js，注入到所有页面（包括 iframe），监听用户的点击和输入事件。

每次用户点击一个按钮，扩展会为这个元素生成一组候选选择器，打包发送给 background script 存储。录制结束后，所有操作导出为一个 JSON 文件。

回放端：Python 桌面应用

基于 Playwright（异步 API）+ PySide6 GUI。加载 JSON 脚本，启动浏览器，逐条回放操作。遇到视频自动等待播完，遇到 iframe 自动穿透，出错可以跳过或停止。

同时提供了 CLI 和 GUI 两种使用方式 —— 命令行派和图形界面派都能找到自己的舒适区。

核心技术：7 层优先级智能定位

这是我自认为项目里最精彩的部分，也是 v2.0 最大的升级。

旧版录制只会存一个 XPath，比如 /html/body/div[2]/div[1]/div[3]/a[2]。这种绝对路径脆得像饼干 —— 网页加个 <div> 包裹，或者换个顺序，XPath 就废了。

新版的做法是：为每个元素同时生成多种定位方式，按稳定性从高到低排列，回放时依次尝试。

回放时，先试 ID 选择器 —— 找到了直接用，又快又准。找不到？试文本选择器。还找不到？继续往下试。完整 XPath 排在最后，是最不稳的兜底方案。

这就像找人的策略：先查身份证号（ID），再查姓名（文本），再查工位号（data 属性），最后查”左边第三个工位”（绝对路径）。前面的方式更稳定，后面的更通用。

视频检测：不是所有点击都值得等

回放时遇到一个点击操作，怎么判断它是不是”点开视频”？如果误判，要么傻等一个不存在的视频（浪费时间），要么错过视频播放（操作不完整）。

我的方案是双重判断：

  1. 静态判断：看选择器的特征 —— 包含 video、play、vjs、ananas（学习通视频特征路径）等关键字，或者 selectorType 标记为 video-button
  2. 动态判断：点击后，在 8 秒内轮询所有 frame，检测是否有 <video> 元素开始播放

如果 8 秒内检测到视频开始，就自动进入”等视频播完”模式，每秒检查 ended 状态直到完成。如果 8 秒内没有视频开始，就当作普通点击继续下一步。

这套逻辑的关键是超时不能太长 —— 你不想因为一个普通按钮点击而白白等 30 秒吧。

打包：一场与 PyInstaller 的搏斗

把 Python 项目打包成 exe 的过程，就像跟一个脾气不好的合作者谈判 —— 它总能在你最意想不到的地方给你找麻烦。

最坑的一个问题：typing 和 pathlib 这两个包。Python 3.9 已经内置了它们，但 PyPI 上还有旧版的 backport 包。如果用户的 Python 环境里装了这些 backport，PyInstaller 打包时会优先用 PyPI 版本而非内置版本，然后运行时就炸了。

新旧对比：到底进步了多少

旧版（雨课堂专用） 新版（录制-回放通用）
适用范围 仅雨课堂 任意网课平台
换课成本 改源码 重新录制一次
定位策略 单一 XPath（硬编码） 7 层优先级候选选择器
DOM 容错 无 XPath 智能修复（4 条规则）

从”只能开一扇门的钥匙”进化到了”万能钥匙坯 —— 你拿它去比划一下锁的形状，它就能复制出对应的钥匙”。

使用方法

  1. 安装浏览器扩展：Chrome 打开 chrome://extensions/，开启开发者模式，加载 extension1 文件夹
  2. 录制操作：打开网课页面，点击扩展图标 → 开始录制，像正常一样操作网课，完成后停止录制，保存 JSON 文件
  3. 回放：运行 Python 应用，选择 JSON 脚本，点击开始 —— 程序会自动打开浏览器、等待你登录，然后忠实地重放你录制的每一步操作

后记

这个项目最初只是一个不想手动刷课的懒人写的脚本，最后演变成了一个”录制-回放”的通用框架。过程中最深的体会是：与其让程序适配网站，不如让程序适配人。网站千变万化，但人的操作习惯是稳定的 —— 人点按钮就是点按钮，不会因为网站改版就换个操作方式。

所以，把人的操作记录下来再回放，本质上是用”人的适应性”来对抗”网站的不确定性”。这不是什么高深的设计哲学，只是一个刷课刷烦了的人想出来的偷懒办法而已。

但话说回来，人类文明的大多数发明，不都是从偷懒开始的吗？
