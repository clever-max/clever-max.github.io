给宿舍空调加个"遥控器"——从逆向微信小程序到 Flutter 独立应用

一、缘起：室友的半夜抱怨

室友在群里发了一条求助消息——怎么给空调定时？

宿舍的共享空调只能通过微信小程序手动开关。半夜想起床关空调？对不起，请亲自爬起来操作手机。想从教室提前开好空调？不行，红外遥控要求手机和空调在同一房间，而且大部分手机根本没有红外发射头。

这个抱怨让我动了一个念头：既然微信小程序能控制空调，说明厂商有一个云端 API。如果我能搞清楚这个 API 是怎么调用的，就可以写一个自己的 App —— 想要定时？直接在 Android 系统级闹钟上注册，到点自动发 HTTP 请求。想要远程？只要有网络就行，不需要红外硬件。

恰好我也想练练 Flutter 开发，这个项目正合适。

二、第一步：搞清楚”遥控”是怎么实现的

2.1 逆向微信小程序

要复刻一个遥控器，首先得知道原版是怎么工作的。我用 GitHub 上的工具把微信小程序的 wxapkg 包解包，拿到了前端源码。

逆向的结果意外地简单——这个小程序的远程控制完全不需要登录：

整个遥控链路的核心只有三个 HTTP 接口：

接口 作用
GET /device/direct/check?imei=xxx 检查设备是否存在
GET /device/direct/state?imei=xxx 读取空调当前状态
POST /device/direct/command 下发控制命令
手机 App
发送 HTTP 请求 → 厂商云端
红外网关 → 宿舍空调
执行指令
手机不需要红外硬件，厂商云端充当"红外网关" —— 手机发 HTTP，云端转发红外信号给空调

这解释了为什么微信小程序里没有登录就能控制空调——扫码拿到 IMEI 后直接跳过用户认证走直连接口。

2.2 最小可行版本

借助 AI 编程工具，我很快做出了第一个可运行的 Flutter 版本。它做的事很简单：

  1. 模拟小程序的 HTTP 请求（连 User-Agent 都伪装成微信的格式）
  2. 把返回的空调状态展示在屏幕上
  3. 点按钮就下发命令

这个版本跑通的那一刻，我对着手机点了一下”开机”，宿舍空调真的响了。

三、扫码攻坚战：微信扫得开，我的 App 为什么不行？

3.1 问题浮现

MVP 版本有一个致命短板：我只能先在电脑上登录微信小程序、用抓包工具拿到设备 IMEI，然后手动输入到 App 里。这显然不能给别人用。

空调机身上贴有二维码，微信扫码就能直接跳转小程序并绑定设备。理论上，如果这个二维码内容就是 https://xxx?imei=设备号，那我的 App 只要用 ZXing 库扫出来就行。

我用抓包获得的 API 地址反向生成了一枚标准黑白 QR Code，来和空调上的原码做对比：

测试方法 结果
微信扫原码 ✅ 直接打开小程序，自动绑定
QQ 扫原码 ✅ 解析出 URL，包含 IMEI
纯黑白 QR Code（抓包逆向生成） ✅ ZXing 轻松识别
空调上的原码用 ZXing 扫 ❌ 完全解不开
原码用在线解码网站 ❌ 也解不开

左：我根据抓包拿到的 API 地址反向生成的标准 QR Code —— 所有扫码库都能轻松识别

右：空调上实际贴的二维码 —— 只有微信和 QQ 能扫开，ZXing 和在线解码器全部失败

同一个 URL 生成的标准黑白 QR Code，App 秒识别。但空调上那个”官方码”就是不行。

3.2 原因定位

仔细观察后发现：空调上的二维码中间有一个文字 logo，而且存在美化和低对比度问题。虽然人类肉眼看着很清楚，但这种设计对开源扫码库的定位算法造成了严重干扰。

ZXing 这类通用库对”规范、干净”的 QR Code 很拿手，但遇到带 logo、经过美化的码就容易跪。而微信和 QQ 之所以能扫开，是因为它们内部有腾讯自家的增强算法（OpenCV 开源项目里就包含了一套”WeChatQRCode”模块，被用于 WeChat、QQ、QQ Browser 等产品）。

同是 OpenCV WeChatQRCode，Android 版本能解开而 Python 版本解不开——这个差异很微妙，我猜和模型文件版本、预处理管线的实现细节有关。但结果是确定的：有一个现成的方案能解决问题。

3.4 双引擎兜底策略

我把方案设计成这样：

用户选图（相机拍照 / 相册选图）
▼
ZXing 尝试识别
▼
识别成功 → 解析 IMEI 绑定 | 失败 → OpenCV WeChatQRCode 兜底
▼
兜底成功 → 绑定设备 | 失败 → 提示用户手动输入 IMEI
▼
进入遥控器页面
  • 实时扫码：用 ZXing（响应快，标准码秒识别）
  • 相册识别：ZXing 失败后自动切换到 Android 原生 OpenCV WeChatQRCode（专门对付”花里胡哨”的码）
  • 手动输入：保底方案，直接输入 IMEI

这样既保证了标准码的体验，又解决了官方码的兼容性问题。同时我也保留了手动输入入口——毕竟不是所有人都能顺利扫码。

四、定时功能：从”勉强能跑”到”系统级可靠”

4.1 第一版：Dart Timer

最初我觉得定时开关机很简单——设一个 Timer，到点了发 HTTP 请求就行。

问题很快暴露：

  • App 被后台清理 → 定时失效
  • 手机重启 → 定时丢失
  • Flutter 的 Timer 不保证精准触发

这完全是”演示级”的定时，不能当真用。

4.2 第二版：Android AlarmManager + BroadcastReceiver

既然是 Android，那应该用 Android 的方式。我新增了一套原生定时模块：

1 用户在 App
设置 23:30 关机→ 2 Flutter 调原生
schedulePowerAlarm()→ 3 注册系统闹钟
setAlarmClock()→ 4 到点唤醒
AlarmReceiver→ 5 前台 Service
POST HTTP 命令→ 6 写入执行结果
下次开 App 确认
闹钟注册在系统 AlarmManager 服务中，App 被杀 / CPU 休眠都不影响触发

这里有一个关键的技术选择：setAlarmClock 而不是 setExact。

setAlarmClock 是 Android 为闹钟类应用设计的特殊 API。它有两大优势：

  1. 绕过 Doze 低电量限制：系统认为”闹钟应用”有正当理由精准触发
  2. 不需要 SCHEDULE_EXACT_ALARM 权限：用户无需手动授权，不会看到”请允许精准闹钟”的提示

再配合前台 Service + WakeLock，即使 CPU 休眠也能保证 HTTP 命令送达。

4.3 兜底机制

即使做到这个程度，仍然有不能 100% 保证的场景：

  • 国产 ROM 的”上滑清理”：部分国产系统会触发 force-stop，强制取消所有闹钟（这是 Android 设计行为，不是 bug）
  • 手机关机/没电

因此我加了三层兜底：

  1. BootReceiver：手机重启后自动重新注册未到期的定时
  2. 打开 App 时检查：如果发现闹钟已触发但不确定是否执行成功，弹窗问用户”是否补发？”
  3. 国产 ROM 引导：App 内置跳转各品牌自启动设置页面的入口（MIUI/ColorOS/OriginOS 等）

五、细节打磨：不放过每一个”感觉不对”

功能跑通之后，我花了不少时间处理那些”看起来不重要但用户一定能感觉到”的问题。

5.1 状态回跳：点了”开机”，800ms 后按钮又跳回”关机”

这是早期版本最恼人的体验问题。根因在于一个竞态：

用户点击开机→ UI 立即显示"开机"
（乐观更新）→ 后台发 HTTP 命令→ 800ms 后刷新状态→ 读到旧状态："关机"
（空调还没执行完）→ UI 跳回"关机" ✗→ 5秒后读到"开机"
（空调终于执行完）→ UI 又变回"开机" ✓ 这就是「状态回跳」—— 点击 → 乐观显示 → 接口返回旧值 → UI 倒退 → 轮询修正

解决方式：引入确认式刷新机制。

  • 发送命令后，不是立即接受接口返回的状态，而是带上”期望值”去校验
  • 轮询时间序列设为 800ms → 1600ms → 3000ms → 5000ms → 7000ms
  • 只有接口状态与期望值匹配时才更新 UI
  • 如果超过 7 秒仍不匹配，最终接受设备端状态并提示用户

这样一来，用户点下按钮后 UI 就不再”抽风”了。

5.2 相机卡死

mobile_scanner 插件在某些手机上报了一个诡异错误：Attempt to invoke virtual method 'java.lang.Class java.lang.Object.getClass()' on a null object reference。

一开始以为是权限问题，反复折腾了几轮后发现是插件底层 CameraX/MLKit 链路的兼容性问题。最终把插件从 mobile_scanner 换成了基于 ZXing C++/FFI 的 flutter_zxing，问题彻底消失。

5.3 深色模式的”意外”

加了深色模式支持后，效果是：

  • 首页：OK，切换流畅自然
  • 控制页：半透明渐变叠加 → 深色背景下某些文字和按钮对比度不够

这个页面的背景是一个彩色渐变（空调开着是暖色，关着是冷色），本身就用透明度+亮度做了层次。在浅色模式下看起来很好，但切换到深色模式后，底层画布变暗，叠加的透明度效果会让文字变得”飘”。

最后的方案是：深色模式换一套更暗的渐变起始色，而不是简单让系统自动适配。

5.4 APK 体积：从 90MB 到 47MB

第一个打包出来的 APK 将近 90MB。一看内容：

  • OpenCV 原生物理库打了 三套 ABI（arm64、armv7、x86_64），仅 OpenCV 就占了 67MB！
  • ZXing 库也自带多 ABI

简单的优化：

  • Release 只打 arm64-v8a（现代手机基本都是 arm64）
  • OpenCV 依赖只引入 opencv-armv64，删掉 32 位和 x86 版本

APK 直接降到 47.5MB。虽然不算小，但对于内部使用来说可以接受了。

七、后续展望

当前版本已经可以日常使用，但还有几个方向值得继续：

  1. 云端定时服务：目前定时依赖手机在线。如果部署一个后端定时服务，App 只负责提交任务，到点由服务端调用厂商 API，手机被清后台、关机都不受影响。
  2. APK 体积进一步压缩：如果确认 OpenCV 兜底足够可靠，可以去掉 ZXing 库，预计能再省 5-10MB。
  3. 多设备管理：目前只能绑定一台空调。宿舍如果有两台，就需要解绑再重新绑定。
